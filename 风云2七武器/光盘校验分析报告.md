# FongYung（风云）免CD补丁分析报告

## 概述

- **游戏名称**: 风云 / FongYung / WindCloud
- **发行年份**: 2001
- **开发商**: 昱泉国际 (InterServ)
- **引擎**: WCCW 2.0 (自研，版本号 0.97WCCW2)
- **可执行文件**: `FongYung.exe` (913,408 字节, PE32, ImageBase 0x400000)
- **CD验证方式**: 光盘类型检测 + 卷标验证 + CD编号检测 + CD文件加载
- **补丁总修改量**: 6处, 共33字节（v1为5处29字节，因栈不平衡bug修复为6处33字节）

---

## 一、与 KnifeMan（小李飞刀）的对比

两款游戏使用**完全相同的自研 WCCW 引擎**，CD验证架构高度一致：

| 对比维度 | KnifeMan (小李飞刀) | FongYung (风云) |
|----------|---------------------|-----------------|
| CD卷标 | "KnifeMan" (8字节) | "FongYung" (8字节) |
| 卷标比较方式 | `memcmp(volName, "KnifeMan", 8)` 直接比对 | `CALL 0xB45C0(volName, fmt)` 子函数比对 |
| CD编号检测 | `CALL ProcessVolumeName(volName)` → 返回3或4 | `CALL 0xB2A5B(volName)` → 返回3或4 |
| 检查成功条件 | `result == 3 \|\| result == 4` | `result == 3 \|\| result == 4` |
| 堆栈帧大小 | 0x278 (632字节) | 0x29C (668字节) |
| CD检查函数RVA | 0x1B5D0 | 0xA6BE0 |
| 对话框系统 | CDialogPlayer 类 | 脚本系统 SCP 内建函数 |
| 已有音频代理 | 需要 _inmm.dll | winmm.dll (用户自加OGG代理，非原始文件) |

## 二、CD验证机制分析

### 2.1 核心CD检查函数 (RVA 0xA6BE0)

```
函数序言:
  0xA6BE0: SUB  ESP, 0x29C      ; 分配栈空间
  0xA6BE6: PUSH EBX,EBP,ESI,EDI
  0xA6BEA: MOV  EBX, ECX        ; this指针 (thiscall)
  0xA6BEC: CALL [GetLogicalDrives]  ; 获取驱动器位掩码
  0xA6BF2: MOV  EDI, 0x41       ; EDI = 'A' (起始盘符)

驱动器枚举循环:
  → SHL  EAX, CL               ; 位掩码测试
  → TEST ECX, EAX
  → JE   NextDrive              ; 驱动器不存在 → 下一个

  → sprintf(buf, "%c:\\", 'A' + index)
  
  检查1: 驱动器类型
  → CALL [GetDriveTypeA]
  → CMP  EAX, 5                ; DRIVE_CDROM
  → JNE  NextDrive              ; ★ 补丁1 — NOP跳转

  检查1.5: 盘符存储
  → MOV  EAX, [EBX+4]          ; 获取之前存储的盘符
  → TEST EAX, EAX
  → JNE  skip
  → MOV  [EBX+4], EDI          ; 存储当前盘符

  检查2: 卷信息
  → CALL [GetVolumeInformationA]
  → TEST EAX, EAX
  → JE   NextDrive              ; ★ 补丁2 — NOP跳转

  检查3: 卷标匹配（不同于KnifeMan！）
  → LEA  ECX, [ESP+0x44]       ; 卷标缓冲区
  → PUSH 2
  → PUSH ECX
  → PUSH "%c:\..."             ; 格式字符串
  → CALL 0xB45C0               ; 字符串比较/解析函数
  → TEST EAX, EAX
  → JNE  NextDrive              ; ★ 补丁3 — NOP跳转

  检查4: CD编号检测
  → CALL 0xB2A5B               ; CD编号检测 (返回3=盘1, 4=盘2)
  → MOV  EBP, EAX              ; ★ 补丁4 — 改为 MOV EBP, 3
  → CMP  EBP, 3
  → JE   store_drive
  → CMP  EBP, 4
  → JNE  skip_store
  → MOV  [EBX+4], EDI          ; 存储盘符

  → CMP  EBP, 3
  → JNE  fail_return            ; ★ 补丁5 — 改为 JMP fail_return (强制成功路径)

CD文件加载 (已通过补丁5跳过):
  → sprintf(buf, "%c:\\resource.dat", drive)
  → CreateFile / ReadFile ...
  → 检查 game.dat, video/*.avi 等CD文件
  
成功返回:
  → MOV  [EBX+0xC], EBP        ; 存储CD编号到对象 → result = 3
  → MOV  [EBX+0x8], EBP        ; 备份存储
  → RET

NextDrive:
  → INC  EDI                   ; 下一个盘符
  → CMP  drive_index, 0x20     ; 最多26个驱动器
  → JL   loop_start
  → RET                         ; 未找到CD
```

### 2.2 与 KnifeMan 的关键差异

**KnifeMan 的补丁3** 是将 `TEST EAX, EAX` (memcmp 返回值测试) 改为 `XOR EAX, EAX` (1字节修改)，利用 `XOR` 同时清零寄存器和设置 ZF=1 的特性。

**FongYung 没有直接 memcmp**，而是调用了两个子函数:
1. `0xB45C0(volName, fmt)` — 字符串比对（可能是 `strnicmp` 或自定义比较）
2. `0xB2A5B(volName)` — CD编号识别（从卷标判断是盘1还是盘2）

因此 FongYung 的补丁3 需要 NOP 掉字符串比对失败后的跳转（6字节），而非 KnifeMan 的 1字节 XOR 技巧。

## 三、补丁方案

### 3.1 补丁汇总

| # | RVA | 原始字节 | 修改字节 | 指令变化 | 作用 |
|---|-----|---------|---------|---------|------|
| 1 | 0xA6C3C | `0F 85 7D 01 00 00` | `90 90 90 90 90 90` | JNE→NOP | 任意驱动器类型→当作CD-ROM |
| 2 | 0xA6C80 | `0F 84 39 01 00 00` | `90 90 90 90 90 90` | JE→NOP | 卷信息获取失败→继续 |
| 3 | 0xA6C9C | `0F 85 1D 01 00 00` | `90 90 90 90 90 90` | JNE→NOP | 卷标不匹配→继续 |
| 4a | 0xA6CA6 | `52 E8 AF BD 00 00` | `B8 03 00 00 00 90` | PUSH+CALL→MOV+NOP | 跳过卷标检测，强制EAX=3 |
| 4b | 0xA6CAE | `83 C4 04` | `90 90 90` | ADD→NOP | 与4a配套保持栈平衡 |
| 5 | 0xA6CC1 | `0F 85 14 01 00 00` | `E9 15 01 00 00 90` | JNE→JMP | 跳过CD文件加载→直接存结果返回 |

### 3.2 补丁详细说明

**补丁1 (0xA6C3C)**: `JNE 0xA6DBF` → `NOP×6`
- 原始: `GetDriveTypeA` 返回后检查是否 `== DRIVE_CDROM (5)`，不是则跳转到下个驱动器
- 修改: NOP 跳转，任意类型驱动器（C:、D: 等）都当作CD-ROM处理

**补丁2 (0xA6C80)**: `JE 0xA6DBF` → `NOP×6`
- 原始: `GetVolumeInformationA` 返回0（失败）时跳转到下个驱动器
- 修改: NOP 跳转，即使无法获取卷信息也继续

**补丁3 (0xA6C9C)**: `JNE 0xA6DBF` → `NOP×6`
- 原始: 卷标字符串比对失败时跳转
- 修改: NOP 跳转，任意卷标都接受

**补丁4a (0xA6CA6)**: `PUSH EDX; CALL 0xB2A5B` → `MOV EAX,3; NOP`
- 原始: 压入卷标指针，调用CD编号检测子函数（返回3=盘1, 4=盘2）
- 修改: 直接 MOV EAX,3，跳过子函数调用，无栈操作

**补丁4b (0xA6CAE)**: `ADD ESP,4` → `NOP×3`
- 4a已移除PUSH EDX，此处的栈清理不再需要，NOP掉保持栈完全平衡

**补丁5 (0xA6CC1)**: `JNE 0xA6DDB` → `JMP 0xA6DDB; NOP`
- 原始: CD编号≠3时跳转到 fail_return
- 修改: 无条件跳转到存储结果并返回（跳过CD文件加载循环）
- 跳转目标计算: 0xA6CC1 + 5 + 0x115 = 0xA6DDB

### 3.3 补丁策略对比

| 维度 | KnifeMan (7处47字节) | FongYung (6处33字节) |
|------|---------------------|---------------------|
| 驱动检查绕过 | NOP JNE (同) | NOP JNE |
| 卷信息失败 | NOP JE (同) | NOP JE |
| 卷标验证 | XOR取代TEST (1字节) | MOV EAX,3+NOP (6字节) — 子函数调用差异 |
| 栈平衡处理 | 不影响栈 | 补丁4a+4b配套（关键！） |
| 强制返回成功 | 32字节代码重组 | 跳过调用+JMP重定向 |
| 堆栈偏移修正 | 有 (1字节) | 不需要 |
| 启动对话框 | NOP CALL (5字节) | 不需要 |
| 游戏内对话框 | JE→JMP (6字节) | 不需要 |

**为什么 FongYung 不需要对话框补丁？**

FongYung 的 CD检查函数始终返回成功 (result=3)，调用方检查 `[obj+0xC] == 3` 得到真值，自动跳过所有 CD对话框代码。KnifeMan 需要单独 NOP 对话框是因为旧补丁未完全覆盖调用方的检查路径。

### 3.4 调试教训：v1 栈不平衡导致的启动崩溃

**现象**: 补丁 v1 版启动后进程加载到 ~33MB RAM，随即报错降到 2MB 后退出。

**根因**: 原补丁4 将 `MOV EBP,EAX; ADD ESP,4` (5字节) 替换为 `MOV EBP,3` (5字节)。但 `ADD ESP,4` 的作用是清理之前 `PUSH EDX` 压入的4字节——删掉它后 ESP 永久偏移 +4，函数末尾 `ADD ESP,0x29C; RET` 弹出错误的返回地址，进程崩溃。

**教训**:

| 问题 | 说明 |
|------|------|
| 栈操作必须成对 | PUSH 和 ADD ESP 是配对的操作，替换其中一个必须处理另一个 |
| `MOV EBP,3` 不够 | 5字节指令换了5字节看起来"正好"，但语义完全不同——少了一次栈清理 |
| 反汇编不能只看指令 | `8B E8` (MOV EBP,EAX) 本身无害，但和后面的 `83 C4 04` 是逻辑整体 |
| 子函数调用链 | 原代码 `PUSH → CALL → MOV → ADD` 四步是一个完整的"调子函数取返回值"模式，不能只改中间两步 |

**修复**: 拆为 4a+4b — 4a 将 `PUSH EDX; CALL` 替换为 `MOV EAX,3; NOP`（不碰栈），4b 将 `ADD ESP,4` NOP 掉。九字节零栈操作，完全平衡。

**排查过程**: 最初怀疑注册表缺失（`reg.reg` 中有 `HKLM\...\InterServ\WindCloud2` 键），但通过导入表分析确认 FongYung.exe 根本不导入 `ADVAPI32.dll`，无注册表 API 调用，排除了此可能性。

## 六、技术要点

### 6.1 FongYung vs KnifeMan — 同引擎不同实现

两款游戏使用相同的 WCCW 引擎，CD验证的四层架构（驱动检查→类型检查→卷标检查→文件验证）完全一致。但在卷标比对环节：

- **KnifeMan**: 直接使用 C运行时库的 `memcmp(volName, "KnifeMan", 8)` + `TEST EAX,EAX` + `JE`
- **FongYung**: 调用自定义函数 `CALL 0xB45C0` 做字符串处理，可能涉及字符集转换或更复杂的卷标解析

这反映了同一个引擎在不同项目中的定制化程度——核心框架复用，但具体实现细节随游戏需求调整。

### 6.2 CD编号系统

FongYung 的 CD编号检测函数 (0xB2A5B) 从卷标解析 CD 编号:
- 返回 3 → 第一张游戏盘
- 返回 4 → 第二张游戏盘

游戏启动和换盘逻辑通过此编号区分不同光盘内容。补丁强制返回 3 意味着游戏始终认为第一张盘在驱动器中。

### 6.3 winmm.dll OGG代理（用户自加，非原始文件）

`fongyung/winmm.dll` 是用户自行添加的自定义 Windows Multimedia DLL 代理（编译于 2025年1月），将游戏的 MCI CD音频命令重定向到 OGG 文件。这解决了 CD音轨播放问题，但不涉及光盘验证。两者相互独立。
